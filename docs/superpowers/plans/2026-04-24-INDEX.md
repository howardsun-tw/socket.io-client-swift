## Execution

For each phase:
1. Branch from `master`: `git checkout -b phase-N-<topic>`
2. Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to execute the plan.
3. Open PR, merge, then proceed to next phase.

Phases 1, 2, 3, 6 are fully independent — can ship in parallel. 4, 5, 7, 8, 9 should ship in order.

## Cross-cutting test conventions

Per spec test parity rule (`feedback_test_parity.md`):
- Every Swift unit test that mirrors a JS reference test must reference the JS test name in a comment.
- Swift-only edge cases (concurrency, identity swap, namespace isolation, v2/v3 parity) are listed in each plan's testing section.
- E2E tests use the existing `TestServerProcess` fixture and may require server-side fixture additions (documented per phase).

## JS-divergence policy

All Swift-side innovations are documented in the spec under one of 5 categories:
1. Swift-idiomatic mappings (Phase 2 `.error` channel; Phase 9 enum; Phase 4/5 UUID handles)
2. Swift concurrency overloads (Phase 8/9 async)
3. Stricter Swift API (Phase 7 no volatile+ack)
4. Preserved Swift legacy (Phase 1/5/9 `.error` on disconnected emit; Phase 9 legacy emitWithAck not cleared)
5. Swift-side error channels where JS is silent (Phase 8 v2 + async-throw)

If implementing a phase reveals a divergence not in one of these categories, escalate before merging.
